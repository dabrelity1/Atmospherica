## 2024-10-24 - Coordinate Mapping Traps in Vector Optimization
**Learning:** The codebase constructs `Vec3(x, z, time)` in weather calculations, mapping the Z coordinate to the vector's Y component and Time to Z. When unpacking `Vec3` to primitives for optimization, `pos.y` does not always correspond to vertical position.
**Action:** Always trace `Vec3` constructor arguments `(x, y, z)` to their semantic meaning before replacing with primitives, especially when `Vec3` is used as a generic data container.
## 2024-10-24 - Early-Exit Wind Damage Evaluation
**Learning:** In `GameBusEvents.onLevelTick`, wind damage logic involves an intense loop simulating check points around valid players up to 260 times per player to determine if blocks should break or move based on wind vectors. This evaluation heavily relies on `WindEngine.getWind`, which allocates memory and consumes CPU even when there is no possibility for damage (ambient wind speeds don't surpass the block damage thresholds).
**Action:** When a global check (e.g. `weatherHandler.getStorms().isEmpty()`) can determine that a heavy evaluation phase is irrelevant, wrap the heavy process directly in that check. Make sure only the logic explicitly depending on that state is skipped so ambient generation patterns (e.g. ambient wind force application for player movement) or future feature evaluations are not incorrectly broken.
